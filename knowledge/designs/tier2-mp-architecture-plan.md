> **⚠️ HISTORICAL (v1) — superseded.** This is the original Tier 2 plan. It
> describes the **v1** design that compiles/runs against `UnlicensedAdapter`
> and `aria-ops-core.jar` — both **retired** in the v2 framework (which
> eliminated the `aria-ops-core` dependency). Read this for history only.
> Current architecture: `context/tier2_architecture.md`; current v2 design:
> `designs/vcfcf-base-v2-adapterbase-rehome.md` + the per-adapter migration

## Context

The VCF Content Factory's Tier 1 MPs (MPB-based) are early beta. Tier 1 limits have been hit concretely: Cloudflare (no dynamic timestamps, JMESPath filter gaps) and Synology (data model doesn't fit MPB's chained-collection pattern). Tier 2 adds native Java SDK adapters that can run custom collection logic — the thing MPB fundamentally can't do.

The cleanroom bundle (`vcf-mp-cleanroom-bundle-2026-05-16.zip`) provides a comprehensive spec of the SDK contract, the adapter lifecycle, and a framework design guide. This plan turns that spec into a concrete implementation.

**Goal:** A VCF SME clones this repo, asks for a Synology MP, and gets a working `.pak` — same as Tier 1, but with custom Java code under the hood. The user needs a JDK installed; everything else is automated.

## Architecture

### Four-layer stack (from cleanroom spec §17)

```
Layer 4: Per-adapter code (Claude-generated, ~50-150 lines)
         SynologyAdapter extends VcfCfAdapter<SynologyConfig>
         Config POJO, API calls, response mapping
─────────────────────────────────────────────────────
Layer 3: vcfcf-adapter-base.jar (framework, pre-compiled)
         VcfCfAdapter<C> extends UnlicensedAdapter
         Auth, HTTP, retry, pagination, describe.xml builder
─────────────────────────────────────────────────────
Layer 2: aria-ops-core.jar (Broadcom, shipped as-is)
         UnlicensedAdapter, Tester/Discoverer/Collector SPI
─────────────────────────────────────────────────────
Layer 1: vrops-adapters-sdk.jar (Broadcom, on appliance)
         AdapterBase, AdapterInterface3
```

### Build approach: Python-driven with `javac`

- **No Maven, no Gradle, no pom.xml.** The Python tooling calls `javac` directly, manages the classpath from JARs shipped in the repo, and assembles the `.pak`. Same pattern as Tier 1 where Python handles everything.
- **JDK is the only external dependency** for Tier 2 (Tier 1 stays zero-dependency). The tooling detects `javac` on PATH, gives clear guidance if missing, and can offer to help install it.
- **Framework JAR is pre-compiled** by us, committed to the repo under `src/vcfops_managementpacks/adapter_runtime/`. Same pattern as `mpb_adapter-9.0.1-patch-1.jar` today.
- **Dependency JARs are shipped in the repo** under `adapter_runtime/` — the SDK, aria-ops-core, HTTP client, JSON parser. No dependency resolution needed; the classpath is deterministic.
- The `build` command detects Tier 1 (YAML) vs Tier 2 (Java project in `content/sdk-adapters/`) and routes accordingly.

### Key decisions in this structure

- **`content/sdk-adapters/`** — parallel to `content/managementpacks/`. Each subdirectory is a self-contained adapter project. No pom.xml.
- **`adapter.yaml`** — lightweight metadata file (like the top of a Tier 1 YAML but without requests/objects). Tells the builder this is Tier 2, what the adapter_kind is, version, dependencies.
- **Framework JARs in `adapter_runtime/`** — pre-compiled, version-pinned. The same directory that holds the Tier 1 MPB runtime today.
- **No Maven wrapper, no Gradle wrapper** — `javac` + our Python tooling is the build system.

## Tier Decision Framework

Added to `mp-designer` agent and documented in `knowledge/context/tier_decision_framework.md`:

**Default: Tier 1** (MPB/YAML) unless any of these triggers are present:

| Trigger | Example | Why Tier 2 |
|---|---|---|
| Non-HTTP protocol | JDBC, SNMP, gRPC, binary | MPB is HTTP-only |
| Complex data model stitching | Synology (multi-call join, session state) | MPB chaining can't express arbitrary joins |
| Stateful collection | WebSocket subscriptions, event streams | MPB is stateless request/response |
| Advanced auth | OAuth2 refresh, Kerberos, mTLS, AWS SigV4 | MPB supports Basic/Token/Custom only |
| Programmatic actions | Restart service, trigger backup | MPB has no action support |
| Dynamic time injection | APIs requiring computed timestamps | MPB has no time variables |
| Cursor-based pagination | Token-in-body pagination | MPB supports offset/page only |
| Custom transforms | Regex extraction, arithmetic, conditional logic | MPB supports BASE64/NONE only |

**Flow:** mp-designer interviews → evaluates triggers against API map → proposes tier with reasoning → user confirms → orchestrator routes.

## Agent Changes

### Updated: `mp-designer`

- Interview now includes tier evaluation step
- Output: design artifact tagged with `tier: 1` or `tier: 2`
- Tier 2 designs include: object model, Java class structure sketch, auth strategy, collection approach, describe.xml outline
- New section: "Promotion reasoning" (why Tier 1 can't handle this)

### New: `sdk-author`

- **Input:** Approved Tier 2 design artifact + API map
- **Output:** Complete Java source under `content/sdk-adapters/<name>/`
- **Generates:** adapter.yaml, all .java files, describe.xml, resources.properties
- **Knows:** the framework API (`VcfCfAdapter<C>`, auth strategies, pagination patterns, metric push helpers)
- **Validates:** compiles successfully via `javac` before reporting done

### New: `sdk-builder`

- **Input:** A Tier 2 adapter project directory
- **Output:** `.pak` file in `dist/`
- **Handles:** JDK detection, `javac` invocation, JAR assembly, adapters.zip construction, pak packaging
- **Validates:** pak-compare against SDK reference paks

### Updated: Orchestrator (CLAUDE.md)

New workflow pattern added:

```
Tier 2 MP: clarify → recon → api-cartographer → mp-designer (proposes Tier 2)
  → user confirms → sdk-author → validate/compile → sdk-builder
  → pak-compare → confirm → content-installer
```

Serial authoring rule still applies (sdk-author before sdk-builder).

## CLI Changes (`vcfops_managementpacks`)

### New commands

| Command | What |
|---|---|
| `build-sdk <dir>` | Compile + package a Tier 2 adapter project into a .pak |
| `scaffold-sdk <name>` | Generate empty Tier 2 project skeleton in `content/sdk-adapters/<name>/` |
| `validate-sdk <dir>` | Check adapter.yaml, compile source, validate describe.xml |

### Updated commands

| Command | Change |
|---|---|
| `build` | Auto-detect: if arg is `.yaml` → Tier 1; if arg is a directory with `adapter.yaml` → route to `build-sdk` |
| `validate` | Include Tier 2 projects in validation sweep |
| `list` | Show both Tier 1 and Tier 2 MPs with tier label |

### New module: `sdk_builder.py`

Core logic (pure Python + `javac`/`jar` subprocess calls, no Maven):
1. Load `adapter.yaml` metadata (name, version, adapter_kind, build_number)
2. Detect JDK: `javac -version` on PATH; clear error + install guidance if missing
3. Build classpath: `adapter_runtime/vcfcf-adapter-base.jar` + `adapter_runtime/aria-ops-core-*.jar` + `adapter_runtime/vrops-adapters-sdk-*.jar` + project `lib/*.jar`
4. Find sources: `glob src/**/*.java`
5. Compile: `javac -cp <classpath> -d <build_dir> <sources>`
6. Generate `adapter.properties` (ENTRYCLASS + KINDKEY) from adapter.yaml
7. Package adapter JAR: `jar cf <name>.jar -C <build_dir> .` (includes adapter.properties)
8. Copy describe.xml, resources, icons into adapter conf directory
9. Collect runtime deps for `lib/` (framework JAR + HTTP client + JSON — NOT the SDK JARs which are on the appliance classpath)
10. Assemble adapters.zip (SDK pak structure — no design.json, no template.json)
11. Generate manifest.txt (SDK format — no `adapters:` field, empty script slots)
12. Write outer .pak ZIP
13. Run pak-compare against SDK reference paks (HPE SimpliVity, Pure Storage)

## Framework JAR: `vcfcf-adapter-base.jar`

### Scope (Phase 1 — minimum for Synology)

| Component | What |
|---|---|
| `VcfCfAdapter<C>` | Abstract base extending `UnlicensedAdapter`. Typed config binding, lifecycle defaults. |
| `HttpClientBuilder` | Fluent builder for HTTP client with base URL, SSL config, default headers |
| `AuthStrategy` | Interface + implementations: Basic, Bearer, SessionCookie (Synology uses SYNO.API.Auth session tokens) |
| `RetryPolicy` | Exponential backoff with jitter. Default: 3 attempts, retry on 5xx/429/IOException |
| `MetricPusher` | One-liner helpers: `push(resource).metric(key, value)`, `.property(key, value)` |
| `DescribeBuilder` | Typed builder for describe.xml covering ResourceKinds, CredentialKinds, Identifiers, Attributes |

### Deferred (add when needed by a second adapter)

- OAuth2 client-credentials auth
- Kerberos / SPNEGO
- AWS SigV4
- Cursor-based pagination
- Link-header pagination
- Cross-MP attachment helpers
- Relationship fluent builder
- Action annotation dispatcher
- Capacity/Policy describe.xml elements

### Build process

The framework JAR is built **once by us** with a JDK and committed to the repo. Users never compile it. A simple `build-framework.sh` script in `src/vcfops_managementpacks/adapter_framework/` handles the rebuild when we change framework source:

```bash
# One-time framework build (we run this, not users)
javac -cp adapter_runtime/aria-ops-core-*.jar:adapter_runtime/vrops-adapters-sdk-*.jar \
  -d build/ adapter_framework/src/**/*.java
jar cf adapter_runtime/vcfcf-adapter-base.jar -C build/ .
```

Framework source lives at: `src/vcfops_managementpacks/adapter_framework/src/` (Java source for the framework itself). No Maven, no Gradle — just `javac` + `jar`.

## Synology as First Adapter

**Why it was deferred from Tier 1:** The Synology DSM API requires session-based auth (SYNO.API.Auth), multi-call joins across unrelated API endpoints (storage volumes + system info + Docker containers), and hierarchical data stitching that MPB's chained-collection model can't express.

**API maps already exist:** 8 files in `context/api-maps/synology-*.md` covering auth, storage, system, Docker, iSCSI, UPS, events, and overview.

**Adapter structure (estimated):**
- `SynologyConfig.java` — host, port, credentials, API version
- `SynologyAuth.java` — implements `AuthStrategy` for SYNO.API.Auth session tokens
- `SynologyDiscoverer.java` — discovers volumes, disks, pools, Docker containers
- `SynologyCollector.java` — collects metrics per resource type
- `model/` — response POJOs (Volume, Disk, SystemInfo, DockerContainer, etc.)
- `describe.xml` — 5-8 resource kinds (NAS, Volume, Disk, Pool, DockerContainer, etc.)

**Estimated per-adapter code:** ~200-400 lines (slightly above the 50-150 target due to Synology API quirks, but well within "Claude generates, user reviews" scope).
