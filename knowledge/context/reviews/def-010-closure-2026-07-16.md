# DEF-010 closure — live compute proof + UI-vs-API discrepancy (2026-07-16)

**Verdict: DEF-010 CLOSES.** With the `net|packets*` denominator keys
policy-enabled, the "ESXi Bad Network Packets" SM computes on hosts and
its cluster rollup computes on clusters. The formula is sound; the defect
was policy enablement, now proven end-to-end and documented in the pak.

## Evidence chain (all 2026-07-16, devel)

1. **Catalog recon:** `net|packetsRx/Tx_summation(_sum)` exist in the
   VMWARE HostSystem catalog, `default_monitored: false`; the collecting
   `dropped*/errors*` siblings are `true`
   (`knowledge/context/investigations/recon_log.md`, dated entry).
2. **Keys collect when enabled:** policy-enabled 20:39Z; by 21:49Z all
   four keys carried real accumulating values on the test host
   (`packetsRx_summation_sum` 326202.0, `packetsTx_summation_sum`
   251501.0).
3. **SM computes (leaf):** UI chart on `vcf-lab-mgmt-esx03`, SM
   `ESXi Bad Network Packets` (`sm_c0c98494-1147-4ec1-affd-fcaf54334563`,
   identity confirmed via API name lookup): nonzero datapoints at
   15:53:49 (9.999e-5) and 16:14:12 (1.604e-4) local — inside the
   enablement window. User independently observed the same before the
   browser pass.
4. **SM computes (rollup):** UI chart on the mgmt cluster,
   `vSphere Cluster Worst ESXi Bad Network Packets`
   (`sm_48f81e75-bbc4-45f2-860f-4f0683d2a56d`): multiple nonzero points
   15:50–16:50 local, peak ~1.8e-4.
5. **Remediation shipped:** pak PR #8 documents the post-install
   policy-enablement requirement (SMs + the two `net|packets*`
   attributes). The metrics were re-enabled on devel 2026-07-16 evening
   and left enabled (user decision) — the chain now computes permanently
   there.

## The "UI-vs-API discrepancy" that wasn't (FB-016, resolved same day)

Every API check that day queried the bare statkey `sm_<uuid>` and got
`{"values": []}` — but the stats surface keys SM series as
**`Super Metric|sm_<uuid>`**. Discovered by listing
`/api/resources/{id}/statkeys` (both SMs were plainly present, prefixed)
after a plain-metric control proved the endpoint itself worked. Re-swept
with the correct key (03:0xZ, POST `/api/resources/stats/query`, 8h
window):

| Tier | Result |
|---|---|
| Hosts (9/9) | Leaf SM series present on all — 73 pts from 20:43Z (4 min after enablement); mgmt-esx03 last 8.2e-5, mgmt-esx04 2.05e-5, others computed 0 |
| Clusters (3/3) | Rollup series present on all — mgmt-cl01 last 8.2e-5 (worst-host), others 0 |

API values match the UI charts exactly. The three earlier "SM not
computing" verdicts (21:51Z single-host, 21:57Z fleet sweep, and the
first post-re-enable check) were all false negatives from the bare-key
query — factory-side error, no platform bug. Codified as
`knowledge/lessons/sm-statkey-api-prefix.md`; FB-016 reclassified
accordingly (the FB-010 policy-API 500 on the same instance is unrelated
and remains open).

Screenshot artifacts were session-local and are not retained in the repo
(convention); the observed values and timestamps are recorded above.
